# ADR 0004: 계정 삭제와 재가입 정책

## 상태

채택됨 (구현 예정)

## 맥락

notegate는 개인 PII를 직접 다루는 B2C 서비스이며, de-identification(식별 차단)을 의도한다.
따라서 계정 삭제는 단순 비활성화가 아니라 데이터가 실제로 소거되는 흐름이어야 한다.

현재 동작에는 다음 문제가 있다.

- 탈퇴 직후 식별 연결을 끊어 버려서, 같은 신원으로 다시 로그인해도 이전 계정을 잇지 못하고
  **매번 새 계정/고아 계정**이 생긴다. "돌아온 사용자" 처리가 사실상 동작하지 않는다.
- 삭제 side effect가 자동으로 광범위해, 공유 workspace 멤버의 접근이 **조용히** 사라진다.
- 즉시 완전 삭제는 리소스 부담이 크고, 실수로 인한 삭제를 되돌릴 여지가 없다.

ADR 0002는 attribution 보존을 위해 account를 일반 product action으로 hard delete하지 않으며,
workspace/node는 soft delete 후 purge한다고 정했다. 본 ADR은 계정 삭제 lifecycle을 이와 정합하게
구체화한다.

## 결정

### 1. 삭제는 영구이며, 재가입은 삭제가 완전히 완료된 뒤에만 가능하다

- 탈퇴한 계정은 부활하지 않는다.
- 삭제 프로세스가 끝나기 **전에는** 같은 신원(OAuth sub)으로 재가입하거나 사용할 수 없다. 이 동안의
  로그인은 거부한다(쿨다운).
- 삭제가 **완전히 완료된 뒤** 같은 신원으로 다시 오면 **새 계정**이 된다. 이전 데이터와는 연결되지
  않는다.
- 삭제는 취소할 수 없다. 진행 중(완료 전) 상태에서 되돌리는 경로를 제공하지 않는다.

### 2. 하이브리드 정리 — workspace는 사용자가 명시적으로, 나머지는 자동

- **단독 소유 workspace**: 사용자가 직접 삭제하거나 소유권을 이전한다. 소유한 workspace가 남아
  있으면 탈퇴를 차단한다(전제조건).
- **공동 workspace(다른 active user owner 존재)**: 탈퇴 시 자동 leave하며 workspace는 생존한다.
- **agent / api key**: 탈퇴 시 자동 정리한다.

### 3. 즉시가 아니라 유예 후 완료한다

- 삭제는 즉시 완전 소거하지 않고, 주기적 purge로 일정 기간 내에 완료한다.
- 이는 즉시 hard delete의 리소스 부담을 줄이고 일괄 처리하기 위함이며(지연 삭제), workspace/node의
  기존 soft delete → purge 흐름과 일관한다. **복구(undo)를 위한 창이 아니다 — 삭제는 되돌리지 않는다.**
- 구체 기간(완료 상한)과 purge 주기는 `docs/spec/lifecycle.md`가 정본이다.

### 4. 식별 정보는 삭제가 완료될 때 제거하고, attribution은 익명 형태로 보존한다

- 식별 PII는 삭제 프로세스가 **완료(완전 소거)되는 시점**에 제거한다. 완료 전(쿨다운) 동안에는
  접근이 차단된 상태로 유지되며, 외부에서 사용할 수 없다.
- 소거 후에도 과거 기여물의 attribution 무결성을 위해 **식별 불가능한 익명 형태**는 남길 수 있다.
  이는 ADR 0002의 "account를 일반 action으로 hard delete하지 않는다"와 정합한다.

## 라이프사이클 (개념)

```text
[활성] ──탈퇴──▶ [삭제 진행 중] ──같은 신원 로그인──▶ ⛔ 거부 (아직 완료 전)
                     │
                 (유예 경과 · 완전 소거)
                     ▼
                [삭제 완료] ──같은 신원 로그인──▶ ✅ 새 계정 (이전 데이터와 무관)
```

## 근거

- 영구 삭제 + 완료 후 재가입은 de-identification 의도와 PIPA 파기 원칙에 정합한다. 재가입을 위한
  식별 연결을 영구 보관하지 않는다.
- "완전 소거 후에만 재가입"이 중복 계정과 고아 누적을 원천 차단한다.
- 주기적 purge로 완료하면 즉시 hard delete의 리소스 부담을 없애고 일괄 처리할 수 있다. 복구 창은
  제공하지 않는다 — 삭제는 영구다.
- workspace 수동 정리는 공유 멤버에게 발생하는 깜짝 데이터/접근 손실을 막는다. 삭제는 명시적
  사용자 의도여야 한다.
- 익명 attribution 보존은 ADR 0002의 attribution 원칙을 깨지 않으면서 PII만 제거한다.

## 결과

- 탈퇴는 전제조건(소유 workspace 정리)을 만족해야 시작된다.
- 삭제~완료 사이 같은 신원의 로그인은 거부된다.
- 완료 후 같은 신원은 이전과 무관한 새 계정으로 시작한다.
- 브라우저/세션 인증과 API key 인증은 삭제 진행/완료 상태를 동일하게 취급한다.
- 삭제 side effect, anonymization 시점, 유예/purge 단계의 구체 정의는 `docs/spec/lifecycle.md`와
  `docs/spec/security.md`가 정본이다.
- audit log/event history는 별도 ADR에서 정의한다.
